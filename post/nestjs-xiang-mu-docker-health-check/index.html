NestJs 项目 docker health check

2026-04-30 · 3 分钟阅读

前几天发现 nestjs 打包的 docker 项目内存占用非常巨大

排查步骤总结

我通过以下步骤和命令逐步排查出 warehouse-api 容器内存占用大的根本原因（健康检查误用导致大量重复进程）：

  1. 检查容器基本状态
    命令：docker ps --filter id=9f377f5a286d --format "table {{.ID}}\t{{.Names}}\t{{.Image}}\t{{.Status}}\t{{.Ports}}"
    发现：容器状态为 Up 30 minutes (unhealthy)，端口映射正常。

  2. 检查内存和 CPU 使用
    命令：docker stats --no-stream --format "table {{.Container}}\t{{.Name}}\t{{.MemUsage}}\t{{.MemPerc}}\t{{.CPUPerc}}" 9f377f5a286d
    发现：内存使用 5.053GiB / 13.54GiB (37.31%)，CPU 10.82%。

  3. 检查容器内进程列表
    命令：docker exec 9f377f5a286d sh -c "ps -o pid,comm,rss,vsz,pcpu | head -20"
    发现：多个 node 进程，每个 RSS 140-180MiB，VSZ 10g。

  4. 检查容器详细配置和健康检查
    命令：docker inspect --format='ID: {{.Id}}\nImage: {{.Config.Image}}\nCmd: {{.Config.Cmd}}\nEntrypoint: {{.Config.Entrypoint}}\nEnv: {{range .Config.Env}}{{println .}}{{end}}\nHost Memory: {{.HostConfig.Memory}}\nHealth: {{if .State.Health}}{{.State.Health.Status}} ({{range .State.Health.Log}}{{println .Output}}{{end}}){{else}}none{{end}}' 9f377f5a286d
    发现：健康检查 CMD node dist/main，多次超时失败；内存限制为 0（无上限）。

  5. 检查进程父子关系
    命令：docker exec 9f377f5a286d sh -c "ps -o pid,ppid,comm,args | grep node | head -40"
    发现：所有 node 进程都是 node dist/main，PPID 显示为 1 或其他进程。

  6. 查找配置文件
    工具：file_search 查询 **/{Dockerfile,docker-compose*.yml,*.sh,package.json}
    发现：Dockerfile 和 package.json。

  7. 读取 Dockerfile 和 package.json
    工具：read_file 读取 Dockerfile 和 package.json
    发现：Dockerfile 中健康检查 HEALTHCHECK ... CMD node dist/main；package.json 无多进程启动脚本。

  8. 搜索代码中的 worker/cluster 相关
    工具：grep_search 查询 cluster|worker_threads|fork\(|spawn\(|isPrimary|isMaster|Worker 在 src/**
    发现：只有 @nestjs/bullmq 的 WorkerHost（队列处理器），非 Node 集群模式。

  9. 检查进程运行时间
    命令：docker exec 9f377f5a286d sh -c "ps -o pid,ppid,etime,comm,args | grep 'node dist/main' | head -40"
    发现：进程 etime 依次递减（间隔 ~30s），匹配健康检查周期。

通过这些步骤，确认问题不是代码中的 worker，而是健康检查配置错误导致的进程累积。
